# RemindMe — Этап 3: Надёжность

## Current State

После этапа 2 напоминания создаются, отображаются и доставляются базовым циклом worker (`scheduled`→`sending`→`sent`), отменяются через сервис. Нет повторных попыток при сбоях отправки, восстановления зависших записей при старте, команды `/timezone`, специальных DST-проверок и централизованной обработки ошибок для пользователя.

## Description

Добавить надёжность доставки и времени: повторные попытки с эскалацией до `failed`, восстановление после перезапуска, смену часового пояса пользователя, отклонение несуществующего/неоднозначного локального времени и единые тексты ошибок.

## Scope

**In scope:**
- Worker: повторные попытки по расписанию 30/120/600 секунд, инкремент `attempt_count`, обновление `next_attempt_at_utc`; после 4-й неудачи статус `failed`. Распознавание блокировки бота пользователем как пути к `failed`.
- Восстановление при старте: возврат `sending` с `locked_at_utc` старше 60 с в `scheduled`; немедленная отправка просроченных активных напоминаний независимо от длительности простоя.
- `/timezone <IANA>` — смена `users.timezone` с валидацией через `zoneinfo`; изменение не двигает уже созданные напоминания (меняется только отображение).
- DST-обработка: отклонение несуществующего и неоднозначного локального времени в момент перевода часов (`ZoneInfo`/`fold`); расширение парсера типизированной ошибкой «несуществующее/неоднозначное время».
- Централизованный обработчик ошибок aiogram: `SQLAlchemyError` → откат → текст «Не удалось выполнить действие. Попробуйте ещё раз немного позже.».
- Текст «Напоминание уже отправляется и не может быть отменено.» (отмена записи, уже переведённой worker в `sending`).
- Structured-логирование через стандартный `logging` (structured `extra`, без новой зависимости — по `conventions.md`): `user_id`, тип сущности, `attempt_count`, код ошибки, длительность; без текста записей и токена.
- `setMyCommands` добавляет `/timezone`.

**Out of scope:**
- Exactly-once доставка — MVP держит at-least-once; дубли при аварии между отправкой и фиксацией статуса допустимы и не исправляются.
- Заметки, задачи, inline-кнопки (этап 4).
- Docker Compose, end-to-end smoke (этап 5).

## Acceptance Criteria

- **Восстановление после перезапуска:** напоминание, сохранённое до остановки, после старта приложения и наступления времени находится worker и доставляется.
- Симуляция ошибки `send_message`: первая неудача → `attempt_count=1`, `next_attempt_at_utc = now+30`; после 4-й неудачи → `failed`.
- Зависшее `sending` (`locked_at_utc` > 60 с) при старте возвращается в `scheduled` и затем доставляется.
- `/timezone Europe/Moscow` меняет зону; `/timezone Mars/Phobos` отклоняется; уже созданное напоминание не меняет UTC-момент, но его отображение пересчитывается.
- Несуществующее локальное время (в день весеннего перевода) и неоднозначное (осеннего) отклоняются с просьбой указать другое время.
- Ошибка SQLite при сохранении → транзакция откатывается, пользователь видит «Не удалось выполнить действие. Попробуйте ещё раз немного позже.»; лог содержит код ошибки без текста записи.
- Отмена напоминания, уже переведённого worker в `sending`, возвращает «Напоминание уже отправляется и не может быть отменено.»
- В логах нет значения токена и полного текста записей (тест с перехватом логов).

## Risks and Constraints

- Развязка повторы/восстановление требует аккуратной работы со временем; текущий момент в тестах передаётся явно, реальный `sleep` не используется.
- Интеграционные тесты блокировки и повторов worker должны проверять атомарность захвата записи (паттерн `UPDATE … WHERE status='scheduled'` из `sqlalchemy.md`).

## Scope Estimate

Single task. Третья подзадача серии.

## Existing Architecture

Зависит от этапа 2: рабочий цикл worker, парсер, репозитории напоминаний, статусная модель. Расширяет `worker/notifications.py`, `bot/handlers.py` (`/timezone`, общий error handler), `services/parser.py` (DST-ошибки), `db/repositories.py` (восстановление `sending` при старте).
